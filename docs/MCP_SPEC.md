# MCP_SPEC.md
## MCP Server Specification — Tools, Contracts, and Execution Environment

---

### Execution Environment Decision

**Choice: Restricted Python subprocess + DuckDB in-process**

Rationale:
- Docker adds setup overhead incompatible with a 24-hour build
- Subprocess is honest, fast, and sufficient for Python/Pandas/SQL tasks
- DuckDB handles all SQL execution including window functions with zero server setup
- Two separate execution paths (Python vs SQL) keep the MCP clean and testable

Security constraints on the subprocess:
- Hard timeout: 10 seconds via `signal.SIGALRM` (Unix) or `threading.Timer` (Windows)
- `ast.parse()` safety check before execution — reject obvious injection attempts
- No network calls permitted in submitted code (block `import requests`, `import urllib`, `import socket`)
- No filesystem writes permitted (block `open(..., 'w')`)
- Execution in a temporary directory that is deleted after each run

---

### MCP Server Entry Point

The server runs locally and exposes tools over stdio (standard MCP transport).

```
gauntlet/mcp/server.py
```

Started automatically by `main.py` as a subprocess before the LangGraph session begins.

### Tool 1: `execute_python_with_tests`

**Purpose**: Run candidate's Python/Pandas code against hidden unit tests generated by the QA Engineer.

**Input schema**:
```json
{
  "user_code": "string — the candidate's submitted Python code",
  "test_cases": [
    {
      "test_id": "string — unique identifier e.g. 'test_null_handling'",
      "setup_code": "string — any imports or data setup needed before the test",
      "assertion_code": "string — the actual assert statement(s)"
    }
  ],
  "timeout_seconds": "integer — default 10"
}
```

**Output schema**:
```json
{
  "exit_code": "integer — 0 if all tests pass, 1 if any fail, 2 if timeout, 3 if syntax error",
  "tests_passed": "integer",
  "tests_total": "integer",
  "results": [
    {
      "test_id": "string",
      "passed": "boolean",
      "error_type": "string or null — e.g. 'AssertionError', 'KeyError'",
      "traceback": "string or null"
    }
  ],
  "execution_time_ms": "float",
  "stdout": "string — any print() output from user code",
  "stderr": "string"
}
```

**Execution logic**:
1. Run `ast.parse(user_code)` — if SyntaxError, return `exit_code: 3` immediately
2. Check for blocked imports — if found, return `exit_code: 3` with descriptive stderr
3. For each test case, execute: `setup_code + "\n" + user_code + "\n" + assertion_code`
4. Capture result per test
5. `exit_code = 0` only if all tests pass

**Example call** (from QA Engineer):
```python
tool_call = {
    "user_code": "def rolling_avg(df):\n    return df.groupby('user_id')['minutes'].rolling(7).mean()",
    "test_cases": [
        {
            "test_id": "test_output_shape",
            "setup_code": "import pandas as pd\ndf = pd.DataFrame({'user_id': [1,1,1], 'minutes': [10, 0, 5]})",
            "assertion_code": "result = rolling_avg(df)\nassert result is not None"
        },
        {
            "test_id": "test_null_handling",
            "setup_code": "import pandas as pd\ndf = pd.DataFrame({'user_id': [1,1,None], 'minutes': [10, None, 5]})",
            "assertion_code": "result = rolling_avg(df)\nassert result.isna().sum() == 0, 'Nulls not handled'"
        }
    ]
}
```

### Tool 2: `execute_sql_with_tests`

**Purpose**: Run candidate's SQL query against a DuckDB in-memory database and validate the result.

**Input schema**:
```json
{
  "user_query": "string — the candidate's submitted SQL",
  "schema_setup": "string — DuckDB SQL to create and populate test tables",
  "test_cases": [
    {
      "test_id": "string",
      "validation_query": "string — SQL that returns a boolean or raises if incorrect",
      "description": "string — what this test checks"
    }
  ],
  "timeout_seconds": "integer — default 10"
}
```

**Output schema**: Same structure as `execute_python_with_tests`.

**Execution logic**:
1. Create in-memory DuckDB connection
2. Execute `schema_setup` to build test tables
3. Execute `user_query` and store result as a temp view: `CREATE VIEW candidate_result AS (...user_query...)`
4. For each test case, execute `validation_query` against `candidate_result`
5. A test passes if `validation_query` returns a truthy result without raising

**Example call**:
```python
tool_call = {
    "user_query": "SELECT user_id, date, AVG(minutes) OVER (PARTITION BY user_id ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) as rolling_7d FROM listening",
    "schema_setup": """
        CREATE TABLE listening (user_id INT, date DATE, minutes FLOAT);
        INSERT INTO listening VALUES
            (1, '2024-01-01', 30), (1, '2024-01-02', NULL), (1, '2024-01-03', 45);
    """,
    "test_cases": [
        {
            "test_id": "test_window_present",
            "validation_query": "SELECT COUNT(*) > 0 FROM candidate_result WHERE rolling_7d IS NOT NULL",
            "description": "Rolling average column exists and has values"
        },
        {
            "test_id": "test_null_days",
            "validation_query": "SELECT COUNT(*) = 0 FROM candidate_result WHERE minutes IS NULL AND rolling_7d IS NULL",
            "description": "Days with null minutes are still included in rolling calc"
        }
    ]
}
```

### Error Codes Reference

| Code | Meaning | Hiring Manager Response |
|---|---|---|
| 0 | All tests pass | Proceed to Judge |
| 1 | One or more assertions failed | Show traceback, offer feedback |
| 2 | Execution timeout | "Your solution exceeded the time limit — consider time complexity" |
| 3 | Syntax error or blocked import | "There's a syntax issue before we could run tests" |

---

### What the MCP Does NOT Do

- It does not suggest fixes
- It does not explain what the correct answer is
- It does not communicate with the user directly
- It does not persist any state between calls

All interpretation of MCP results is handled by the Hiring Manager agent.
